# Claude Code 源码解析 - 记忆系统

## 文章大纲

### 一、引言
- 记忆系统在 AI 驾驭中的核心地位
- Claude Code 记忆系统的设计目标：持久化、上下文感知、跨会话学习

### 二、记忆类型系统：四类记忆的精妙设计

> 来源：`src/memdir/memoryTypes.ts`

#### 2.1 四种 Memory Type
- **user** — 用户角色、目标、偏好、知识
- **feedback** — 用户指导（纠正 + 确认）
- **project** — 项目上下文、正在进行的工作
- **reference** — 外部系统指针（Linear、Grafana 等）

#### 2.2 核心设计原则
- 记忆只保存**不可从代码推导**的信息
- 明确排除：代码模式、Git 历史、调试方案等
- **feedback 要包含 Why 和 How to apply**
- 相对日期必须转换为绝对日期

#### 2.3 记忆的「遗忘」机制
- 记忆会过时（memory drift）
- 使用前必须验证
- 用户可以要求「忽略」记忆

### 三、记忆存储架构：文件系统的艺术

> 来源：`src/memdir/paths.ts`

#### 3.1 目录结构
```
~/.claude/projects/<sanitized-git-root>/memory/
├── MEMORY.md          # 入口索引
├── user_role.md       # 用户记忆
├── feedback_xxx.md     # 反馈记忆
├── project_xxx.md     # 项目记忆
└── reference_xxx.md   # 引用记忆
```

#### 3.2 多层路径解析
1. `CLAUDE_COWORK_MEMORY_PATH_OVERRIDE` 环境变量
2. `settings.json` 中的 `autoMemoryDirectory`
3. 默认：`~/.claude/projects/<project>/memory/`

#### 3.3 安全设计
- 路径验证防止 `..` 遍历
- 不支持 `~` 根路径
- `projectSettings` 不能设置 `autoMemoryDirectory`（防恶意仓库）

### 四、团队记忆：多人协作的支持

> 来源：`src/memdir/teamMemPrompts.ts`

#### 4.1 双目录架构
- **private memory** — 个人记忆，仅自己可见
- **team memory** — 团队记忆，项目成员共享

#### 4.2 scope 机制
- user → always private
- feedback → default private, team only when project-wide convention
- project → strongly bias toward team
- reference → usually team

### 五、记忆召回：智能相关性检索

> 来源：`src/memdir/findRelevantMemories.ts`

#### 5.1 召回流程
1. 扫描 memory 目录
2. 读取 frontmatter 获取 description
3. 使用 Sonnet 模型选择相关记忆
4. 返回最多 5 个相关记忆

#### 5.2 避免噪音
- 排除 MEMORY.md（已在 system prompt）
- 最近使用的工具不显示其 reference docs
- 但保留 warnings、gotchas、known issues

### 六、记忆压缩：上下文空间的守护者

> 来源：`src/services/compact/compact.ts`

#### 6.1 压缩时机
- **自动压缩**：当上下文接近 token 限制时
- **手动压缩**：`/compact` 命令
- **部分压缩**：压缩特定消息段

#### 6.2 压缩流程
1. 调用 LLM 生成对话摘要
2. 保留最近的上下文（文件状态、工具结果）
3. 用摘要替换旧消息
4. 重新注入必要的附件（文件、技能、计划）

#### 6.3 压缩策略
- **文件恢复**：保留最近读取的 5 个文件
- **技能保留**：保留调用的技能（最多 25K tokens）
- **图片剥离**：用户图片不参与摘要生成

#### 6.4 压缩边界标记
- 创建 `__COMPACT_BOUNDARY__` 标记
- 摘要作为用户消息注入
- 支持后续的增量压缩

### 七、设计目标：为什么这样设计？

#### 7.1 核心设计哲学
1. **持久化优先于检索** — 记忆应该持久化，而不是每次重建
2. **可验证性** — 记忆可以被验证和更新
3. **隐私分层** — 支持 private 和 team 不同层级
4. **简洁性** — 文件系统是最简单的记忆存储

#### 7.2 解决什么问题？
- **上下文窗口有限**：记忆系统解决跨会话信息保留
- **Agent 健忘症**：Agent 无法记住之前对话中学到的信息
- **团队协作**：多人需要共享上下文
- **上下文膨胀**：长时间对话会超出 token 限制

#### 7.3 什么不该做？
- 记忆不等于日志
- 记忆不等于文档
- 记忆只保存「非显而易见」的信息
